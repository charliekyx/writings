What: 第一篇文章具体要做什么？
这篇文章（Corridor Audit: MYR → USD via IBKR）将通过 MDX 格式，呈现一个完整的资金路由取证过程：

定义摩擦公式： 明确指出真实成本 = 表面手续费（Nominal Fees） + 汇率点差（FX Spread） + 中转行剥削（Intermediary Leaks） + 时间损耗（Time Decay）。

数据快照： 抓取并对比特定时间窗口内，SWIFT 电汇与第三方 Fintech（如 Wise）的实际执行汇率与 Google 中间价的偏离度。

注入交互式组件 (React in MDX)： 在文章核心位置直接嵌入一个 <CrossBorderCalculator />。读者输入本金，组件实时输出不同路径下的“精确到账金额差值（Exact Delta）”。

Why: 为什么必须这么做？(护城河逻辑)
1. 抵御平台规则变更（Anti-Fragility）： 如果只写操作步骤，一旦 IBKR 官方更新本地转账通道，文章就会瞬间沦为垃圾，甚至被社区喷（如之前的 Reddit 经历）。而当你输出的是“测算模型”和“基准数据”时，无论官方怎么更新，你只需把新通道的数据喂进公式，你依然是评估这个新通道是否真诚的“独立审计官”。

2. 建立极客信任溢价（The Independent Audit Moat）：
使用第一人称（"I write code to track..."）和冰冷的数据，能立刻劝退伸手党，筛选出高净值交易员和量化开发者。
这些人对“暗箱操作”极度反感，当你用代码把黑盒拆开，他们会心甘情愿使用你内置的 Affiliate 链接，甚至未来为你的 API 付费。

的沉浸体验： 当一个吉隆坡的 IP 访问这篇文章，MDX 里的 <CrossBorderCalculator /> 会默认将基准货币设为 MYR；而当新加坡的 IP 访问时，虽然看的是同一个组件逻辑，但默认环境会自动切换为 SGD 相关的参数。这把一篇普通的博客，变成了一个**“具有本地执行上下文的私人金融终端”**。
